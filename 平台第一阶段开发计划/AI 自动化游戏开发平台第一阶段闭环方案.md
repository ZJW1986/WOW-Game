# AI 自动化游戏开发平台第一阶段闭环方案

## Summary

第一阶段建设一个 Web 端 AI 游戏生成平台的最小闭环，参考 [Rosebud.ai](https://rosebud.ai/) 的 `Create + Play` 产品结构，但核心逻辑采用“标准产物流水线”：用户从一句自然语言开始，经由引导问答、策划文档、资源需求、游戏逻辑、试玩版本、资源库归档、Play 发布、反馈迭代，完成第一个可体验小游戏。

第一阶段优先验证平台流程，不追求真实全模型生产。默认使用模拟模型与内置资源跑通闭环，同时预留真实 LLM、生图、生音效、BGM、生特效模型适配器。

## Key Architecture

- **前端**：`React + TypeScript + Vite`，包含 Studio、Pipeline、Asset Hub、Preview、Play、Project Versions 六个核心视图。
- **游戏运行层**：`Phaser`，第一阶段只生成 2D Web 小游戏原型。
- **后端**：`Node.js + TypeScript`，提供项目、流水线、资源、模型任务、发布、反馈 API。
- **数据层**：第一阶段用 `SQLite` 或本地 JSON 数据存储，表结构按后续数据库迁移设计。
- **资源层**：本地文件资源库，统一管理图片、音效、BGM、特效、游戏构建产物和来源 metadata。
- **模型层**：`Model Provider Adapter`，所有模型能力通过统一任务协议调用，默认启用 Mock Provider。
- **编排层**：`Pipeline Orchestrator`，负责把每个阶段的标准输出传给下一阶段。

## Standard Pipeline

平台第一阶段固定 8 个阶段，每阶段必须输出 `JSON` 机器可读文件和 `Markdown` 人类可读文件：

1. **Idea Intake**
   - 输入：用户一句话创意、目标风格、游戏类型偏好。
   - 输出：创意摘要、目标玩家、核心体验、缺失信息列表。

2. **Guided Design**
   - 输入：创意摘要。
   - 输出：系统自动追问 3-5 个关键问题，引导用户补齐玩法、角色、目标、难度、视觉风格。

3. **Game Design Document**
   - 输出标准 GDD：游戏说明、背景、角色、技能、核心循环、数值、关卡、胜负条件、操作方式、目标平台。

4. **Asset Requirement**
   - 输出标准资源需求：角色图、场景图、UI、道具、音效、BGM、特效、尺寸、风格、数量、用途、版权状态。

5. **Asset Production**
   - 第一阶段默认用内置/占位资源模拟。
   - 真实模型接口预留：生图、音效、BGM、特效。
   - 所有资源进入 Asset Hub，记录来源、生成参数、所属项目、版本、引用关系。

6. **Game Logic Spec**
   - 输出游戏逻辑规格：实体、状态、输入、碰撞、分数、关卡流、失败/胜利条件、数据事件。

7. **Playable Build**
   - Phaser 根据标准配置生成可试玩 Demo。
   - 第一阶段支持固定模板：躲避、射击、收集、平台跳跃、简单解谜，优先实现 1-2 个模板。

8. **Play & Iterate**
   - 发布到 Play 页面，其他玩家可试玩。
   - 收集反馈：评分、文本建议、游玩时长、失败点、重开次数。
   - 反馈进入下一轮迭代建议，生成新版本。

## Model Integration Design

- **LLM/代码模型**
  - 用于生成 GDD、资源需求、逻辑规格、迭代建议。
  - 接口字段：`provider`、`model`、`taskType`、`prompt`、`schema`、`temperature`、`maxTokens`、`responseFormat`。
  - 第一阶段提供 Mock Provider 和真实 Provider 占位配置。

- **生图模型**
  - 任务类型：角色、场景、道具、UI、图标、封面。
  - 输出：图片文件、prompt、negative prompt、尺寸、风格、版权状态。

- **音频模型**
  - 任务类型：音效、BGM、UI 声音、环境声。
  - 输出：音频文件、时长、循环点、用途、生成参数。

- **特效模型**
  - 第一阶段先用预设特效库模拟。
  - 后续可接视频/粒子/序列帧生成模型。
  - 输出：特效类型、触发条件、资源路径、参数。

## Core Product Modules

- **Studio**
  - 用户输入自然语言想法。
  - 平台以向导方式追问关键问题。
  - 展示当前阶段、产物状态、下一步动作。

- **Pipeline Console**
  - 展示每个阶段的输入、输出、状态、错误、重试。
  - 支持查看标准 JSON 和 Markdown 文档。

- **Asset Hub**
  - 管理图片、声音、BGM、特效、脚本、构建包。
  - 支持按项目、类型、版本、来源筛选。
  - 每个资源必须有 metadata，避免后期版权和引用混乱。

- **Game Preview**
  - 内嵌 Phaser 试玩窗口。
  - 支持重新生成、调整参数、保存版本。

- **Play**
  - 展示已发布小游戏。
  - 玩家可试玩、评分、反馈。
  - 每个游戏有版本号、封面、简介、作者、玩法说明。

- **Iteration Center**
  - 汇总玩家反馈和游玩数据。
  - 生成下一轮改版建议：难度、数值、关卡、美术、音频、玩法节奏。

## Required Skills

建议项目内创建并维护可执行 Skill，后续每次生成游戏时优先加载，减少跑偏：

- `game-production-pipeline`
  - 定义从创意到发布的阶段顺序、输入输出、验收标准。

- `game-design-document`
  - 生成标准 GDD，包括背景、角色、技能、数值、关卡、胜负条件。

- `game-asset-brief`
  - 把 GDD 转换成图片、音效、BGM、特效的标准需求。

- `phaser-game-template`
  - 按标准逻辑规格生成 Phaser 小游戏模板。

- `game-qa-review`
  - 检查可玩性、白屏、报错、资源缺失、胜负流程、操作反馈。

- `game-iteration-advisor`
  - 根据玩家反馈和指标生成下一版迭代建议。

第一阶段可以先把这些 Skill 作为项目内文档和模板维护；等开发执行时，再安装到 Codex skill 目录或项目 `.agents/skills` 目录。

## MVP Acceptance Criteria

- 用户能输入一句游戏想法，并通过引导问答补齐信息。
- 系统能生成标准 GDD、资源需求、逻辑规格。
- 系统能用 Mock 模型和内置资源生成一个可试玩 Phaser Demo。
- 生成的图片、声音、BGM、特效占位资源能进入资源库并被游戏引用。
- 游戏能发布到 Play 页面，其他用户可试玩并提交反馈。
- 项目支持版本记录：至少能区分 `v1` 和一次迭代版本。
- 所有阶段产物都可查看、可复用、可作为下一阶段输入。
- 模型供应商、模型名称、任务类型都通过配置选择，不写死在业务逻辑里。

## Test Plan

- **流程测试**：从一句话创意完整跑通到 Play 页面发布。
- **产物测试**：检查每个阶段都有 JSON 和 Markdown 输出。
- **资源测试**：检查资源入库、metadata、引用关系、缺失资源提示。
- **试玩测试**：打开生成游戏，能开始、游玩、失败/胜利、重新开始。
- **反馈测试**：玩家反馈能写入项目并生成迭代建议。
- **模型测试**：Mock Provider 可稳定返回结果；真实 Provider 缺配置时给出清晰提示。
- **回滚测试**：每个游戏版本可查看历史产物，后续开发阶段再接 Git 提交和回滚。

## Assumptions

- 第一阶段目标是“基础闭环走通”，不是完整商业游戏生产系统。
- 第一阶段默认 2D Web 小游戏，不做 3D、Unity、Godot、Roblox 导出。
- 第一阶段真实多媒体生成模型只预留接口，不作为必须成功路径。
- 第一阶段发布是站内 Play 发布，不自动上架应用商店或外部平台。
- 第一阶段优先保证标准化产物、资源管理和迭代闭环，模型效果可以后续增强。
